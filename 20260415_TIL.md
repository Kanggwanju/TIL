## EAI(Enterprise Application Integration)

- 다양한 앱(시스템)들을 하나로 묶어서,
  데이터가 흐를 수 있도록 만들어주는 `중앙 통합 시스템`

### EAI 구축 유형

1. Point-to-Point

- 시스템 간에 필요한 연결을 직접 1:1로 생성하는 방식

  장점: 둘만 연결하니까 간단하고 빠름

  단점: 연결할 앱이 많아지면 스파게티처럼 얽혀서 관리 안 됨

2. Hub & Spoke

- 중앙 관제탑 방식, 중앙 통제
- 모든 시스템이 관제탑을 통해서만 데이터를 주고받는 방식

  장점
  - 중앙 통제, 관리가 편함, 데이터 흐름 추적 용이

  단점
  - 병목현상: 모든 정보가 허브로 몰리면 막힐 수 있음
  - 단일 실패 지점(SPOF): 허브가 다운되면 전체 시스템이 멈춤

3. Message Bus

- 버스 노선이라는 공용 데이터 통로를 두고,
  각 시스템이 버스 정류장이 되는 방식

  장점
  - Point-to-Point보다 유연
  - Hub&Spoke보다 확장성이 좋음
  - 새로운 시스템 추가 쉬움

  단점
  - 버스 노선에 데이터 몰리면 병목 현상 가능

  - 전체적인 관리가 Hub&Spoke보다 복잡

4. Hybrid

- Hub와 Message Bus 방식 섞어 씀
- 빠르고 안정적인 Hub로 중요 정보 전송
- Bus로 덜 중요한 정보 전송

  장점: 유연하고 각 방식의 장점만 가져옴

  성능: 병목 현상 최소화 가능

  단점: 구조가 복잡, 비용이 많이 든다.

## IPSec

- 인터넷 프로토콜에서 보안성을 제공하는 표준화된 기술
- 네트워크 계층에서 암호화, 인증, 키 관리를 통해 데이터의 기밀성과 무결성을 보장

1. 암호화: IPSec은 데이터 암호화를 통해 통신 내용을 비밀로 유지한다. 이를 통해 중간에서 패킷이 가로채져도 내용을 해독할 수 없게 한다.

2. 인증: 패킷의 출처를 확인하는 기능을 제공하여, 데이터가 신뢰할 수 있는 소스에서 온 것임을 보장한다. 이는 메시지의 무결성과 송신자의 신원 확인에 필수적이다.

3. 키 관리: 안전한 키 교환 메커니즘을 통해 통신 양측이 암호화 및 복호화에 필요한 키를 효과적으로 관리하고 교환할 수 있도록 지원한다. 이는 보안 연결을 유지하는 데 중요한 역할을 한다.

## 정보 보호 기술 `AAA`

1. Authentication

- 시스템을 접근하기 전에 접근 시도하는 사용자의 신원을 검증

2. Authorization

- 검증된 사용자에게 어떤 수준의 권한과 서비스를 허용

3. Accounting

- 사용자의 자원(시간,정보,위치 등)에 대한 사용 정보를 수집

## 네트워크 보안, `세션 하이재킹`

- '세션을 가로채다.' 라는 의미로 다른 사람의 세션 상태를 훔치거나 도용하여 액세스하는 해킹 기법이다.

- TCP `세션 하이재킹`은 TCP의 3-way 핸드셰이크가 완료된 후에 공격자가 시퀀스 번호 등을 조작하여 정상적인 세션을 가로채고 인증 없이 통신을 탈취하는 해킹 기법이다.

## CPU 스케줄링

- 여러 `프로세스`들 중에 누가 먼저 `CPU`를 사용할지 정하는 작업
- `운영체제(OS)`가 수행하는 중요한 역할 중 하나
- `OS`가 `스케줄러`를 통해 결정함

### 스케줄링 알고리즘

- `스케줄러`는 CPU 스케줄링 `알고리즘`을 사용해서
- 어느 `프로세스`를 실행할지 정함

### 스케줄링 방식

1. 비선점 (Non-Preemptive)

- 한번 CPU를 할당받으면, 작업이 끝날 때까지
- CPU를 빼앗을 수 없는 방식

2. 선점 (Preemptive)

- 더 급한 작업이 나타나면, OS가 강제로
- CPU를 빼앗아 다른 프로세스에게 할당할 수 있는 방식

비선점 스케줄링 알고리즘

1. FCFS(First-Come, First-Served)

- 선입 선출
- 준비 큐에 도착한 순서대로 CPU 할당

콘보이 효과

- FCFS의 단점
- 호위 효과(Convoy Effect)
- 실행 시간이 긴 프로세스 하나가 맨 앞에 실행되면
- 그 뒤의 짧은 프로세스들이 전부 기다려야 하는 현상

2. SJF(Shortest-Job-First)

- 가장 빨리 끝낼 수 있는 일을 먼저 처리
- = 실행 시간이 제일 짧은 일을 먼저 처리
- 평균 대기 시간이 가장 짧음

기아 현상(Starvation)

- SJF의 단점
- 실행 시간이 짧은 프로세스가 계속 들어오면
- 실행 시간이 긴 프로세스는 영원히 기다려야함

3. HRN(Highest Response-ratio Next)

- SJF의 기아 현상 해결을 위해 등장
- 대기 시간을 고려하여 `우선순위` 점수를 매김

우선순위 계산식

- 우선순위 = (대기 시간 + 서비스 받을 시간) / 서비스 받을 시간
- 우선순위가 높으면 먼저 실행

선점 스케줄링 알고리즘

1. 라운드 로빈(Round Robin, RR)

- 모든 손님이 공평하게 5분씩만 계산대를 쓰게 하자!
- 시간 할당량(Time Quantum) = 5분
- 시간이 끝나면 자원을 회수하여 다른 프로세스를 실행

시간 할당량이 너무 크면
-> FCFS와 같음
시간 할당량이 너무 작으면
-> 문맥 교환이 너무 자주 일어나서 오버헤드가 커짐

결론: 적절한 시간 할당량을 찾는게 관건

2. SRT(Shortest Remaining Time)

- 최소 잔여 시간 우선 스케줄링
- SJF의 장점과 선점 방식을 합침
- 프로세스를 실행하던 중에 남은 실행 시간이 더 짧은 새로운 프로세스 도착
- -> 문맥 교환, 더 짧은 프로세스에게 자원(CPU)을 할당함

- 새 프로세스가 도착했을 때 남은 시간 비교
