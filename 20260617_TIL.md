# 1. 데이터베이스의 물리 설계와 레코드 접근 방법

데이터베이스의 물리 설계 시, 레코드에 접근하는 방법에는 대표적으로 다음과 같은 방식이 있다.

- 순차 접근 방법
- 인덱스 접근 방법
- 해싱 방법

각 방법은 레코드를 저장하고 검색하는 방식이 다르며, 상황에 따라 장단점이 있다.

---

## 1.1 순차 접근 방법

**순차 접근 방법**은 레코드가 저장된 순서에 따라 처음부터 차례대로 읽어 가며 원하는 레코드를 찾는 방식이다.

이 방식은 구조가 단순하고 구현이 쉽다는 장점이 있다. 또한 모든 레코드를 한 번씩 처리해야 하는 경우에는 효율적이다.

하지만 특정 레코드 하나만 검색해야 할 때는 앞에서부터 순서대로 확인해야 하므로 검색 시간이 오래 걸릴 수 있다.

### 예시

파일에 레코드가 다음 순서로 저장되어 있다고 하자.

```text
A → B → C → D
```

이때 `D`를 찾으려면 `A`, `B`, `C`를 먼저 확인한 뒤에야 `D`에 접근할 수 있다.

### 핵심 정리

| 항목 | 설명 |
|---|---|
| 접근 방식 | 처음부터 차례대로 검색 |
| 장점 | 구조가 단순하고 구현이 쉬움 |
| 단점 | 특정 레코드 검색 시 시간이 오래 걸릴 수 있음 |
| 적합한 경우 | 전체 레코드를 순서대로 처리해야 하는 경우 |

---

## 1.2 인덱스 접근 방법

**인덱스 접근 방법**은 레코드의 키 값과 포인터를 쌍으로 묶어 저장하고, 검색 시 키 값을 기준으로 빠르게 탐색할 수 있도록 설계된 방식이다.

이 방식은 검색 속도가 빠르며, `<키 값, 포인터>` 쌍으로 구성된 자료 구조를 사용한다.

여기서 포인터는 해당 레코드가 저장된 위치를 가리킨다. 따라서 원하는 키 값을 찾으면, 그 키와 연결된 포인터를 통해 실제 레코드가 저장된 주소로 직접 접근할 수 있다.

### 예시

```text
<1001, 주소 A>
<1002, 주소 B>
<1003, 주소 C>
```

키 값 `1002`를 검색하면, 인덱스에서 `1002`를 찾고 연결된 `주소 B`로 이동하여 해당 레코드를 찾는다.

### 핵심 정리

| 항목 | 설명 |
|---|---|
| 접근 방식 | 키 값을 기준으로 인덱스를 검색 |
| 저장 구조 | `<키 값, 포인터>` 쌍 |
| 장점 | 검색 속도가 빠름 |
| 단점 | 인덱스를 위한 추가 저장 공간이 필요함 |

---

## 1.3 해싱 방법

**해싱 방법**은 레코드의 키 값을 해시 함수에 적용하여 해당 레코드가 저장될 주소를 직접 계산하는 방식이다.

이 방식은 해시 함수를 통해 저장 위치를 빠르게 찾을 수 있으므로 검색 속도가 매우 빠르다.

즉, 키 값을 해시 함수에 입력하면 레코드가 저장된 주소가 계산되고, 그 주소를 통해 원하는 레코드에 직접 접근할 수 있다.

### 예시

키 값이 `1001`인 레코드를 찾는다고 하자.

```text
키 값 1001 → 해시 함수 적용 → 저장 주소 계산 → 해당 주소로 접근
```

이처럼 해싱 방법은 키 값을 이용해 저장 위치를 직접 계산한다.

### 핵심 정리

| 항목 | 설명 |
|---|---|
| 접근 방식 | 키 값을 해시 함수에 넣어 주소 계산 |
| 장점 | 검색 속도가 매우 빠름 |
| 단점 | 충돌이 발생할 수 있음 |
| 적합한 경우 | 특정 키 값을 기준으로 빠르게 검색해야 하는 경우 |

---

# 2. 해싱에서 충돌이 발생하는 이유

해싱 방법에서 **충돌**은 서로 다른 키 값이 해시 함수를 거쳤을 때 같은 저장 주소로 변환되는 현상이다.

## 2.1 충돌 예시

해시 함수가 다음과 같다고 하자.

```text
해시 주소 = 키 값 % 10
```

키 값 `1001`의 해시 주소는 다음과 같다.

```text
1001 % 10 = 1
```

키 값 `1011`의 해시 주소도 다음과 같다.

```text
1011 % 10 = 1
```

두 키 값은 서로 다르지만 계산 결과가 모두 `1`이다.

따라서 두 레코드가 같은 주소에 저장되려고 하며, 이때 충돌이 발생한다.

## 2.2 충돌이 발생하는 근본적인 이유

충돌은 다음과 같은 이유로 발생한다.

- 저장 가능한 주소의 수는 제한되어 있다.
- 가능한 키 값의 종류는 저장 주소보다 훨씬 많을 수 있다.
- 해시 함수가 서로 다른 키 값을 같은 주소로 매핑할 수 있다.

즉, 여러 개의 키 값이 한정된 저장 공간으로 변환되기 때문에 충돌이 발생할 수밖에 없다.

---

# 3. 해싱의 충돌 처리 방법

충돌 처리 방법에는 대표적으로 다음 두 가지가 있다.

- 개방 주소법
- 체이닝 방법

---

## 3.1 개방 주소법

**개방 주소법**은 충돌이 발생했을 때 해시 테이블 안에서 다른 빈 주소를 찾아 저장하는 방법이다.

예를 들어 키 값 `1001`이 주소 `1`에 저장되어 있는데, 키 값 `1011`도 주소 `1`로 계산되었다고 하자.

이 경우 주소 `1`에는 이미 데이터가 있으므로, 주소 `2`, `3`, `4`처럼 다른 빈 공간을 찾아 저장한다.

### 개방 주소법의 종류

| 방법 | 설명 |
|---|---|
| 선형 조사 | 충돌이 발생하면 바로 다음 주소를 차례대로 확인한다. |
| 제곱 조사 | 충돌이 발생하면 1², 2², 3²처럼 일정한 간격을 제곱 형태로 늘려 가며 확인한다. |
| 이중 해싱 | 충돌이 발생하면 또 다른 해시 함수를 사용해 다음으로 확인할 주소를 정한다. |

---

## 3.2 체이닝 방법

**체이닝 방법**은 같은 주소로 계산된 레코드들을 하나의 연결 리스트처럼 묶어서 저장하는 방법이다.

예를 들어 키 값 `1001`과 `1011`이 모두 주소 `1`로 계산되면, 주소 `1`에 두 레코드를 함께 연결해 둔다.

이후 주소 `1`을 찾으면, 그 안에 연결된 레코드들을 차례대로 확인하여 원하는 키 값을 찾는다.

### 체이닝 예시

```text
주소 1 → [1001] → [1011]
```

주소는 같지만, 그 주소 안에 여러 레코드를 연결해 저장하는 방식이다.

---

## 3.3 충돌 처리 방법 비교

| 구분 | 개방 주소법 | 체이닝 방법 |
|---|---|---|
| 저장 방식 | 해시 테이블 안의 다른 빈 공간을 찾음 | 같은 주소에 여러 레코드를 연결 |
| 충돌 발생 시 | 다음 후보 주소를 탐색 | 연결 리스트에 추가 |
| 특징 | 테이블 내부에서 해결 | 추가 연결 구조 사용 |

---

# 4. FLSM 방식

## 4.1 FLSM이란?

**FLSM**은 **Fixed Length Subnet Mask**의 약자이다.

FLSM 방식은 하나의 네트워크를 모두 같은 크기의 서브넷으로 나누는 방식이다.

즉, 나누어진 모든 서브넷은 다음 항목이 동일하다.

- 같은 서브넷 마스크
- 같은 IP 개수
- 같은 사용 가능 호스트 수

---

## 4.2 FLSM의 특징

| 항목 | 설명 |
|---|---|
| 서브넷 크기 | 모두 동일 |
| 서브넷 마스크 | 모두 동일 |
| 장점 | 계산과 관리가 단순함 |
| 단점 | 필요한 호스트 수가 서로 다른 경우 IP 주소가 낭비될 수 있음 |
| 적합한 경우 | 같은 크기의 네트워크를 균등하게 나눌 때 |

---

# 5. FLSM에서 비트를 빌리는 이유

FLSM에서 서브넷을 나누려면 기존 호스트 비트 중 일부를 서브넷 비트로 사용해야 한다.

이것을 흔히 **비트를 빌린다**고 표현한다.

서브넷 개수는 빌린 비트 수에 따라 다음과 같이 계산한다.

```text
서브넷 개수 = 2ⁿ
```

여기서 `n`은 빌린 비트 수이다.

---

## 5.1 3개의 서브넷이 필요할 때 왜 2비트를 빌릴까?

`192.168.1.0/24` 네트워크를 FLSM 방식으로 3개의 서브넷으로 나누고 싶다고 하자.

빌린 비트 수에 따라 만들 수 있는 서브넷 개수는 다음과 같다.

| 빌린 비트 수 | 만들 수 있는 서브넷 수 |
|---:|---:|
| 1비트 | 2¹ = 2개 |
| 2비트 | 2² = 4개 |
| 3비트 | 2³ = 8개 |

3개의 서브넷이 필요하지만, 1비트를 빌리면 서브넷이 2개밖에 만들어지지 않는다.

따라서 3개 이상을 만들기 위해 2비트를 빌려야 한다.

2비트를 빌리면 서브넷은 4개가 만들어진다.

즉, FLSM 방식에서는 정확히 3개만 딱 만들어지는 것이 아니라, 같은 크기의 서브넷 4개가 만들어지고 그중 필요한 3개를 사용하는 방식이다.

---

# 6. 예제: 192.168.1.0/24를 FLSM 방식으로 3개로 나누기

## 6.1 문제

```text
192.168.1.0/24 네트워크를 FLSM 방식으로 3개로 나눌 때,
두 번째 서브넷의 브로드캐스트 주소는?
```

---

## 6.2 풀이 과정

### 1단계: 필요한 서브넷 개수 확인

필요한 서브넷 수는 3개이다.

### 2단계: 빌릴 비트 수 결정

서브넷 개수는 `2ⁿ`으로 계산한다.

```text
1비트 → 2개
2비트 → 4개
```

3개 이상이 필요하므로 2비트를 빌린다.

### 3단계: 새로운 프리픽스 계산

기존 네트워크는 `/24`이다.

여기에 2비트를 빌리면 다음과 같다.

```text
/24 + 2 = /26
```

따라서 각 서브넷의 프리픽스는 `/26`이 된다.

### 4단계: 블록 크기 계산

`/26`의 서브넷 마스크는 다음과 같다.

```text
255.255.255.192
```

마지막 옥텟 기준으로 블록 크기는 다음과 같이 계산할 수 있다.

```text
256 - 192 = 64
```

따라서 각 서브넷은 64씩 증가한다.

---

## 6.3 나누어진 서브넷

| 순서 | 네트워크 주소 | 사용 가능한 호스트 범위 | 브로드캐스트 주소 |
|---:|---|---|---|
| 1번째 | 192.168.1.0/26 | 192.168.1.1 ~ 192.168.1.62 | 192.168.1.63 |
| 2번째 | 192.168.1.64/26 | 192.168.1.65 ~ 192.168.1.126 | 192.168.1.127 |
| 3번째 | 192.168.1.128/26 | 192.168.1.129 ~ 192.168.1.190 | 192.168.1.191 |
| 4번째 | 192.168.1.192/26 | 192.168.1.193 ~ 192.168.1.254 | 192.168.1.255 |

---

## 6.4 정답

두 번째 서브넷은 다음과 같다.

```text
192.168.1.64/26
```

이 서브넷의 범위는 다음과 같다.

```text
192.168.1.64 ~ 192.168.1.127
```

여기서 첫 번째 주소 `192.168.1.64`는 네트워크 주소이고, 마지막 주소 `192.168.1.127`은 브로드캐스트 주소이다.

따라서 정답은 다음과 같다.

```text
192.168.1.127
```

# 7. FLSM 말고 다른 방식: VLSM

FLSM 말고 대표적인 방식으로 **VLSM**이 있다.

## 7.1 VLSM이란?

**VLSM**은 **Variable Length Subnet Mask**의 약자이다.

VLSM은 하나의 네트워크를 서로 다른 크기의 서브넷으로 나누는 방식이다.

즉, 각 서브넷마다 필요한 호스트 수에 맞게 서브넷 마스크 길이를 다르게 적용할 수 있다.

---

## 7.2 VLSM 예시

어떤 회사에서 `192.168.1.0/24` 네트워크를 나누는데, 부서별 필요한 호스트 수가 다음과 같다고 하자.

| 부서 | 필요한 호스트 수 |
|---|---:|
| A부서 | 100대 |
| B부서 | 50대 |
| C부서 | 20대 |

FLSM이라면 모든 부서에 같은 크기의 서브넷을 줘야 한다.

하지만 VLSM은 필요한 호스트 수에 맞게 다음처럼 다르게 나눌 수 있다.

| 부서 | 예시 서브넷 | 사용 가능 호스트 수 |
|---|---|---:|
| A부서 | /25 | 126개 |
| B부서 | /26 | 62개 |
| C부서 | /27 | 30개 |

이처럼 호스트가 많이 필요한 곳에는 큰 서브넷을, 적게 필요한 곳에는 작은 서브넷을 할당할 수 있다.

---

## 7.3 FLSM과 VLSM 비교

| 구분 | FLSM | VLSM |
|---|---|---|
| 의미 | 고정 길이 서브넷 마스크 | 가변 길이 서브넷 마스크 |
| 서브넷 크기 | 모두 같음 | 서로 다를 수 있음 |
| 서브넷 마스크 | 모두 동일 | 서브넷마다 다를 수 있음 |
| 장점 | 계산과 관리가 단순함 | IP 주소를 효율적으로 사용 가능 |
| 단점 | IP 주소 낭비 가능 | 계산과 설계가 상대적으로 복잡함 |
| 적합한 경우 | 같은 크기로 균등하게 나눌 때 | 필요한 호스트 수가 서로 다를 때 |

## 데이터베이스 접근 방법

| 방법 | 핵심 개념 |
|---|---|
| 순차 접근 방법 | 처음부터 차례대로 레코드를 검색한다. |
| 인덱스 접근 방법 | `<키 값, 포인터>`를 이용해 레코드 위치를 빠르게 찾는다. |

| 개념 | 설명 |
|---|---|
| 충돌 | 서로 다른 키 값이 같은 해시 주소로 변환되는 현상 |
| 발생 이유 | 키 값은 많고 저장 주소는 제한되어 있기 때문 |
| 처리 방법 | 개방 주소법, 체이닝 방법 |

## 서브넷팅

| 개념 | 설명 |
|---|---|
| FLSM | 모든 서브넷을 같은 크기로 나누는 방식 |
| VLSM | 필요한 크기에 따라 서로 다른 크기로 나누는 방식 |
| 비트 빌리기 | 호스트 비트 일부를 서브넷 비트로 사용하는 것 |
| 서브넷 개수 | 빌린 비트 수가 n일 때 `2ⁿ`개 |
