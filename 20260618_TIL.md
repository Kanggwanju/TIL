## 1. 메서드 오버로딩과 참조 변수 타입의 비밀

### 📌 핵심 개념

- 오버로딩(Overloading)은 **컴파일 시점**에 어떤 메서드를 호출할지 결정하는 정적 바인딩(Static Binding)을 따릅니다.
- 컴파일러는 객체의 실제 런타임 인스턴스 타입이 아니라, 참조 변수의 '선언된 타입'을 기준으로 메서드 시그니처와 매개변수를 매칭합니다.

### ❌ 틀렸던 유형 분석 (`myCar.drive(mySedan)`)

```java
class Car {
    String name = "Car";
    void drive(Car c) {
        System.out.print("Drive-Car:" + c.name + " ");
    }
}

class Sedan extends Car {
    String name = "Sedan";
    @Override
    void drive(Car c) {
        System.out.print("Drive-Sedan:" + c.name + " ");
    }
}

public class Main {
    public static void main(String[] args) {
        Car myCar = new Sedan();
        Sedan mySedan = new Sedan();
        myCar.drive(mySedan); // 결과: Drive-Sedan:Car
    }
}
```

- **동작 원리:** 1. `myCar` 변수의 타입이 `Car`이므로, 컴파일러는 `Car` 클래스에 있는 `drive(Car c)` 호출을 확정합니다.

2. 런타임에 `myCar`는 실제 `Sedan` 객체이므로 오버라이딩된 `Sedan`의 `drive(Car c)`가 실행됩니다.
3. 이때 메서드 내부의 매개변수 `c`는 `drive(Car c)`의 정의에 따라 **`Car` 타입**으로 취급됩니다.
4. **자바에서 멤버 변수(필드)는 다형성이 적용되지 않으므로**, `c.name`에 접근할 때 선언 타입인 `Car` 클래스의 필드 값(`"Car"`)을 가져오게 됩니다.

## 2. 컴파일 시점의 오버로딩 모호성 (Ambiguity)

### 📌 핵심 개념

- 컴파일러는 오버로딩된 메서드 후보군 중 가장 인자와 일치하고 **'더 구체적인(More Specific)'** 메서드를 선택합니다.
- 하지만 인자가 양쪽 메서드의 매개변수 타입에 서로 엇갈리게 부합하면, 어느 한쪽이 우세하다고 판단할 수 없어 컴파일 에러를 발생시킵니다.

### ❌ 틀렸던 유형 분석 (`p.process("Hello", "World")`)

```java
class Processor {
    void process(Object obj, String str) {
        System.out.print("Obj-Str ");
    }
    void process(String str, Object obj) {
        System.out.print("Str-Obj ");
    }
}

public class Main {
    public static void main(String[] args) {
        Processor p = new Processor();
        p.process("Hello", "World");
    }
}
```

- **동작 원리:** \* `"Hello"`와 `"World"` 모두 `String` 타입입니다.
- 첫 번째 메서드 입장에서는 `(String, String)`을 `(Object, String)`으로 받을 수 있습니다.
- 두 번째 메서드 입장에서도 `(String, String)`을 `(String, Object)`로 받을 수 있습니다.
- 컴파일러 관점에서는 두 메서드 모두 일치 확률이 동등하여 결정 장애가 발생하므로 "메서드 호출이 모호합니다(reference to process is ambiguous)"라는 컴파일 에러를 뿜게 됩니다.

## 3. 인터페이스 다운캐스팅과 공변 반환 타입

### 📌 핵심 개념

- **인터페이스 다운캐스팅:** 인터페이스 타입의 참조 변수라 하더라도, 런타임에 가리키는 실제 인스턴스가 특정 클래스 타입이 맞다면 명시적 형변환(`(ClassName) variable`)을 통해 자식 클래스의 고유 멤버에 접근할 수 있습니다.
- **공변 반환 타입(Covariant Return Type):** 자식 클래스에서 메서드를 오버라이딩할 때, 부모 메서드가 선언한 반환 타입의 하위 타입(자식 클래스)으로 반환 타입을 구체화하여 재정의하는 것은 올바른 문법입니다.

### ❌ 틀렸던 유형 분석 (다운캐스팅 & 공변 반환)

```java
// 1. 인터페이스 다운캐스팅
interface Printable {
    void print();
}

class Document implements Printable {
    public void print() {
        System.out.print("Doc ");
    }
    public void preview() {
        System.out.print("Preview ");
    }
}

public class Main {
    public static void main(String[] args) {
        Printable p = new Document();
        p.print();
        if (p instanceof Document) {
            ((Document) p).preview();  // 컴파일 성공 및 정상 실행 (ClassCastException 없음)
        }
    }
}

// 2. 공변 반환 타입
class Master {
    Object getTarget() {
        return new Object();
    }
}

class Apprentice extends Master {
    @Override
    String getTarget() {
        return "Apprentice-Target"; // Object의 자식인 String으로 반환타입 변경 가능!
    }
}

public class Main {
    public static void main(String[] args) {
        Master m = new Apprentice();
        System.out.print(m.getTarget().getClass().getSimpleName());
    }
}
```

## 4. 예외 처리(Catch) 우선순위와 초기화 블록 실행 순서

### 📌 핵심 개념

- **예외 매칭:** 자바의 `try-catch` 구문은 위에서부터 순서대로 `catch` 블록의 타입을 비교하며, **실제 발생한 예외 객체의 런타임 타입**에 가장 먼저 부합하는 단 하나의 블록만 실행합니다. (따라서 하위 예외 클래스를 상위에 배치해야 합니다.)
- **인터벌 초기화 순서:** 객체가 생성될 때 내부 요소들은 아래의 엄격한 순서대로 실행됩니다.
  > **[부모 생성자 `super()`]** ➡️ **[자식 인스턴스 초기화 블록 `{}` 및 필드 명시적 초기화]** ➡️ **[자식 생성자 본문]**

### ❌ 틀렸던 유형 분석 (예외 잡기와 초기화 흐름)

```java
// 1. 예외 처리 구조

class CustomException extends Exception {}
class SubException extends CustomException {}

class HeavyRunner {
    void run() throws CustomException {
        throw new SubException();
    }
}

public class Main {
    public static void main(String[] args) {
        HeavyRunner runner = new HeavyRunner();
        try {
            runner.run();  // CustomException을 상속받은 자식 예외
        } catch (SubException e) {
            System.out.print("SubCatch ");  // 실제 인스턴스가 SubException이므로 여기서 먼저 걸림!
        } catch (CustomException e) {
            System.out.print("CustomCatch ");
        }
    }
}


// 2. 초기화 블록 실행 순서
class InitParent {
    InitParent() {
        System.out.print("Parent-Const ");
    }
}

class InitChild extends InitParent {
    {
        System.out.print("Child-Block ");
    }
    InitChild() {
        System.out.print("Child-Const ");
    }
}
// new InitChild() 실행 결과: Parent-Const -> Child-Block -> Child-Const

public class Main {
    public static void main(String[] args) {
        InitParent obj = new InitChild();
    }
}


class InitChild extends InitParent {
    { System.out.print("Child-Block "); }
    InitChild() { System.out.print("Child-Const "); }
}
```

## 💡 한 눈에 보는 자바 핵심 바인딩 규칙 요약

| 구분                    | 컴파일 타임 (선언 타입 기준 / 정적 바인딩) | 런타임 (실제 인스턴스 기준 / 동적 바인딩)  |
| ----------------------- | ------------------------------------------ | ------------------------------------------ |
| **인스턴스 메서드**     | 호출 가능 여부 및 오버로딩 시그니처 결정   | **오버라이딩된 자식 클래스의 메서드 실행** |
| **정적(static) 메서드** | **선언된 참조 변수 타입의 메서드로 고정**  | 런타임 타입에 영향을 받지 않음 (Hiding)    |
| **멤버 변수 (필드)**    | **선언된 참조 변수 타입의 필드로 고정**    | 다형성이 적용되지 않음                     |
