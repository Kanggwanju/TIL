---

## 1️⃣ String의 핵심: 불변(Immutable)

### 💡 한 줄 정의

**“String은 한 번 생성되면 절대 바뀌지 않고, 변경 시 새로운 객체가 생성된다.”**

---

### 🔍 코드로 확인

```java
String str = "Java";

str.toUpperCase();
str.replace("a", "o");
str.substring(1);

System.out.println(str); // Java
```

👉 모든 메서드는 실행됐지만
👉 **str 자체는 전혀 변경되지 않음**

---

### ✅ 핵심 포인트

- `toUpperCase()` → "JAVA" 반환
- `replace()` → "Jovo" 반환
- `substring()` → "ava" 반환

👉 하지만 전부 **새로운 문자열 반환만 함**

---

### 🎯 결론

👉 **String 메서드 중 “원본을 직접 바꾸는 것”은 없다**

---

## 2️⃣ 문자열을 실제로 바꾸려면

```java
str = str.toUpperCase();
str = str.replace("a", "o");
str = str.substring(1);
```

👉 반드시 **다시 변수에 할당해야 변경됨**

---

## 3️⃣ String vs StringBuilder vs StringBuffer

### 💡 한 줄 정의

- **String** → 불변, 기본값
- **StringBuilder** → 가변, 빠름
- **StringBuffer** → 가변 + 동기화

---

### 🔥 비교 정리

| 구분        | String | StringBuilder | StringBuffer |
| ----------- | ------ | ------------- | ------------ |
| 변경 가능   | ❌     | ✅            | ✅           |
| 속도        | 느림   | 🚀 빠름       | 🐢 느림      |
| 스레드 안전 | ✅     | ❌            | ✅           |

---

### ✅ 언제 쓰냐

#### 👉 1. 일반적인 경우

```java
String name = "Java";
```

👉 대부분은 **String 사용**

---

#### 👉 2. 반복문 / 문자열 누적

```java
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 1000; i++) {
    sb.append(i);
}
```

👉 **StringBuilder 사용 (성능 중요)**

---

#### 👉 3. 멀티스레드 환경

```java
StringBuffer sb = new StringBuffer();
```

👉 거의 안 쓰지만 필요할 때 사용

---

### 🎯 결론

👉 **"문자열 많이 바꾸면 StringBuilder, 아니면 String"**

---

## 4️⃣ == vs equals

### 💡 한 줄 정의

- `==` → **주소 비교**
- `equals()` → **값 비교**

---

### 🔍 예제

```java
String s1 = "BTS";
String s2 = "BTS";
String s3 = new String("BTS");

System.out.println(s1 == s2); // true
System.out.println(s1 == s3); // false
System.out.println(s1.equals(s3)); // true
```

---

### 🎯 결론

👉 문자열 비교는 무조건 `equals()` 사용

---

## 5️⃣ String Pool

### 💡 한 줄 정의

**“같은 문자열을 하나만 생성해서 공유하는 메모리 공간”**

---

### 🔍 구조

```java
String s1 = "BTS";
String s2 = "BTS";
```

👉 하나의 객체를 공유

```
[String Pool]
   "BTS"
   ↑   ↑
  s1  s2
```

---

### ❗ new String()

```java
String s3 = new String("BTS");
```

👉 무조건 새 객체 생성 (Heap)

---

## 6️⃣ intern()

### 💡 한 줄 정의

**“문자열을 String Pool 기준으로 관리하고, 동일한 참조를 반환하는 메서드”**

---

### 🔍 예제

```java
String s1 = "BTS";
String s3 = new String("BTS");
String s4 = s3.intern();

System.out.println(s1 == s4); // true
```

---

### 🔍 동작 원리

1. Pool에 문자열 존재 여부 확인
2. 있으면 → 기존 참조 반환
3. 없으면 → Pool에 저장 후 반환

---

### 🎯 결론

👉 **intern() = 문자열을 Pool 기준으로 재사용하게 해줌**

---

## 7️⃣ null 비교 주의

```java
String s5 = null;

s1.equals(s5); // false (안전)
s5.equals(s1); // ❌ NullPointerException
```

---

### ✅ 안전한 코드

```java
s5 != null && s5.equals(s1)
```

---

## 🎯 최종 정리

| 개념          | 한 줄 요약                      |
| ------------- | ------------------------------- |
| String        | 불변 객체, 변경 시 새 객체 생성 |
| StringBuilder | 가변 객체, 성능 좋음            |
| StringBuffer  | 가변 + 동기화 (거의 안 씀)      |
| ==            | 주소 비교                       |
| equals        | 값 비교                         |
| String Pool   | 동일 문자열 공유                |
| intern        | Pool 기준 참조 반환             |

---

## 💬 오늘의 소감

- String이 왜 변경되지 않는지 직접 코드로 확인하면서 **불변 객체 개념이 확실히 잡혔다.**
- `==`와 `equals()` 차이는 단순 문법이 아니라 **메모리 구조와 연결된 개념**이라는 걸 이해했다.
- `StringBuilder`를 써야 하는 상황(반복문)이 명확해졌고, 실무에서의 기준도 감이 잡혔다.
- `intern()`과 String Pool 개념까지 연결되면서 **자바 문자열 동작 방식이 전체적으로 정리됐다.**
